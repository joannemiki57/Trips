## 0. 한눈에 보기 (TL;DR)

- **무엇**: 여행이 끝난 후, 카메라롤만으로 큐레이션되는 **여행 단위 사진 정리·스토리북 앱** (iOS Native)
- **약속**: *"자동으로 1차 완성, 마음에 들면 직접 다듬기."*
- **1차 타겟**: 인스타 콘텐츠를 만드는 20–30대 여행자
- **확장 타겟**: 여행 사진을 감성적으로 기록·큐레이션하고 싶은 광범위 여행자
- **차별점**: 스토리지 정리 앱의 알고리즘 + 여행 저널 앱의 감성 + **사후 자동 + 사진 원본은 기기 안에**
- **수익 모델**: **Freemium 구독** — 기본 기능(Trip 묶음, favorite, 메모, PDF·카메라롤 export)은 무료, 프로 기능은 구독
- **출시 목표**: 3–4개월 (1인 개발)

## 1. Problem Statement

스토리지 클린업이 아니라, **여행의 감성 기록과 큐레이션**을 위한 사진 정리 앱.

### 1.1 사용자 행동 (관찰)

- 여행에서 사진을 많이 찍음 — 같은 장소·각도에서 여러 장
- 잘 나온 사진과 흔들린 사진이 섞여 있음
- 흔들린 사진 중에서도 감성적으로 좋아하는 컷이 있음
- 여행이 끝나고 나서 정리를 *하긴 해야 하는데* 손이 잘 안 감

### 1.2 풀고 싶은 문제

- 여행 일자별로 자동으로 묶이지 않음
- 중복 사진 처리·마음에 안 드는 사진 처리 과정이 지루함
- "안 예쁜 사진"의 기준이 다양함
  - 풍경: 흔들렸는가
  - 인물: 표정이 이상한가
- 인물만 마음에 안 들지만 풍경/구도는 살리고 싶은 경우 — *AI 인물 수정은 v1 이후로도 영구 보류 (자체 큐레이션 정체성과 충돌)*

### 1.3 기존 갤러리 / 하트 기능의 한계

- 좋아하는 사진을 폴더별로 정리할 수 없음
- 같은 장소에서 찍은 비슷한 사진들을 자동으로 묶어주지 않음
- 흔들렸지만 감성이 좋은 사진을 잘 다루지 못함
- 사진에 컨텍스트(메모·라벨)를 붙일 수 없음
- 아이폰 "추억" 자동 영상은 길고, 중복 미필터, 사용자 통제 부족

### 3.2 Trips 포지셔닝 — 2단 메시지

> 헤드: **"여행이 끝난 뒤, 자동으로 정리되는 프라이빗 여행 스토리북."**
> 서브: **"마음에 안 드는 부분만 손가락으로 살짝 다듬으세요."**

- 1단(자동): 카메라롤에서 trip 자동 묶음 → 중복 그룹핑 → 베스트 1장 자동 추천
- 2단(다듬기): 사용자가 ♥ 다시 고르기·메모 추가·라벨 붙이기로 큐레이션을 완성

## 4. 차별화 전략

**A진영의 알고리즘 + B진영의 감성·수익 + 양쪽의 약점 제거.**

| 경쟁사 약점 | Trips의 해법 |
| --- | --- |
| (저널 앱) 여행 중 수동 업로드 필요 | **사후 카메라롤 자동 처리** |
| (저널 앱) 클라우드 풀업로드 전제 | **사진 원본은 기기에 머문다.** 라벨·메모·favorite 메타데이터만 클라우드 동기화 |
| Apple "추억" — 중복 미필터, 영상 길이 통제 불가 | **중복 그룹핑 후 베스트 1장 + 사용자가 직접 통제** |
| Slidebox / CleanMyPhone — 감성·스토리텔링 X | **여행 단위 큐레이션 + 메모 + export** |

> **데이터 분리 원칙**: *"사진은 당신 기기에 있고, 큐레이션 정보(메모/라벨/favorite)만 동기화됩니다."*
> → 다기기 동기화의 편의 + 사진 프라이버시의 안전을 동시에.

## 5. MVP (v1.0) 핵심 기능

### 5.1 Trip 자동 그루핑

- PhotoKit으로 사진 라이브러리 동기화
- EXIF의 timestamp + location으로 여행 단위 자동 묶음
- 사용자가 시작/종료 날짜 수동 조정 가능

### 5.2 중복 사진 그룹핑

- 같은 시점/같은 위치/유사한 비주얼의 사진을 묶음 (자동 삭제 X, 그룹으로만 표시)
- Live Photo는 Apple 내장 알고리즘 활용해 대표 컷 자동 추천
- 그룹 내에서 사용자가 favorite **자유롭게 지정**(개수 제한 없음)
- favorite 미지정 시: 알고리즘 추천 베스트샷이 임시 대표

### 5.3 메모 + 라벨링 (수동)

- 사진 개별 메모, 또는 그룹 단위 메모
- 라벨 (카페·커피·사람·자연·도시 등) **수동 태깅**
- 라벨 기반 필터/검색
- *AI 자동 라벨링은 v1.5에 추가 (Vision API + Core ML)*

### 5.4 Trip 뷰 — Visual Spine을 기본 뷰로

#### 기본 뷰: **Visual Spine**

- 여행 전체를 **세로 척추(spine)** 으로 압축
- 각 Day가 척추의 마디(node), favorite 한 대표컷만 노출
- favorite이 많은 Day일수록 마디가 굵게/크게 → 큐레이션 밀도 시각화
- 한 화면으로 "이 여행을 내가 어떻게 기억·정리했는지" 파악 가능
- 그대로 **PDF export / 인스타 공유의 골격**으로 사용

```
●─ Day1  [★][★]            (대표 2컷)
│
●─ Day2  [★]               (대표 1컷)
│
●─ Day3  [★][★][★]         (밀도 높음 = 굵은 마디)
│
●─ Day4  (미정리)            (옅게 표시)
```

#### 보조 뷰: 피드 타임라인

- 여행 일자별 세로 스크롤 (인스타 피드 느낌)
- Favorite만 보기 모드 / 라벨별 보기 모드

### 5.5 Export (v1 = 디지털 전용)

- 사진첩 형태 **PDF export**
- 카메라롤 저장 (인스타 업로드 워크플로우용)
- *물리 포토북 인쇄 풀필먼트(Peecho 등 외주)는 v1.5+에 추가*

## 6. UI / UX 명세 (화면 구성)

### 6.1 네비게이션 계층

```
Trips (전체 여행 목록)
  └─ Trip 상세  (예: "Toronto Trip")
       ├─ [기본] Visual Spine 뷰
       └─ [보조] Day별 피드 뷰
            └─ Day 섹션 → 메인 포토 그리드
                 └─ [슬롯 탭] → 중복/유사 사진 클러스터
                       └─ [사진 탭] → Photo Detail (사진 1장 상세 + 메모)
```

### 6.2 화면별 정의

#### Trips — 여행 목록

- 카드형 리스트. 각 카드 = 한 여행
- 카드 썸네일 = 그 여행의 **대표 사진**(favorite 우선)
- 카드 정보: 여행명, 기간(EXIF 자동), 사진 수, Day 수

#### Trip 상세 — 기본은 Visual Spine

- 상단: 여행명 + 기간 + 지도(선택) + 진행도(정리 %)
- 본문: **세로 척추 타임라인** (§5.4 참조)
- 우측 상단 토글로 "피드 뷰"로 전환

#### Day 섹션 (피드 뷰) — 메인 포토 그리드

```
── Day 1 · 6/12 (Mon) · Downtown ───────────
┌────┐ ┌────┐ ┌────┐
│ 1★ │ │ 2  │ │ 3  │     ← 3~4 col 그리드 (장수 제한 없음)
└────┘ └────┘ └────┘
┌────┐ ┌────┐ ┌────┐
│ 4  │ │ 5★ │ │ 6  │     ★ = favorite로 지정된 대표컷
└────┘ └────┘ └────┘
```

- 한 슬롯 = 한 "장면(중복 그룹의 대표 1장)"
- 슬롯 우상단 ♥ = 그 그룹에서 favorite이 정해진 상태
- 메모 있는 슬롯은 작은 메모 아이콘 표시
- 슬롯 모서리 "겹친 카드" 스택 아이콘 = n장 묶임 암시

#### 중복/유사 클러스터 (사진 탭 시)

슬롯 하나를 탭하면 그 장면의 **비슷한 사진 묶음**이 펼쳐진다.

```
[ Day1 · slot 3 의 유사 사진 ]              ✕ 닫기
┌─────────────────────────┐
│        선택된 큰 미리보기          │
│        ( ♥ 누르면 favorite )       │
└─────────────────────────┘
┌──┐┌──┐┌──┐┌──┐┌──┐┌──┐  ← 같은 장면 후보들
│  ││  ││♥ ││  ││  ││  │     (가로 스크롤 스트립)
└──┘└──┘└──┘└──┘└──┘└──┘
   [ 이 장면 메모 추가 ]
```

**핵심 규칙 — favorite ↔ 대표컷 동기화**

- 클러스터 안에서 어떤 사진에 ♥(favorite)를 누르면
- → 상위 그리드의 해당 슬롯 메인 포토 & Visual Spine 마디 둘 다 **즉시 교체**
- 즉, "대표로 보여줄 1장"을 사용자가 묶음 안에서 직접 고르는 구조

#### Photo Detail — 사진 1장 상세 (v1 신규, 2026-05-20 5차 라운드)

클러스터 화면에서 가로 스트립의 사진 1장을 탭하면 진입. Photo.memo 입력 진입점.

```
[ ← 닫기 ]                                  ♥
┌─────────────────────────────┐
│                                            │
│         원본 사진 크게 (탭 = 줌)            │
│                                            │
└─────────────────────────────┘
[ 메모 ]
┌─────────────────────────────┐
│ 이 사진에 대한 메모 (텍스트만, v1)         │
└─────────────────────────────┘
[ 라벨 ] #카페  #서울  +
[ 좌우 스와이프 = 같은 Scene 내 이전/다음 사진 ]
```

- 메모 = 사진별 1개 텍스트 (Photo.memo). Scene 메모와 별개로 보존
- ♥ 우상단 = 클러스터에서와 동일하게 Scene.representativePhoto 즉시 교체
- 좌우 스와이프로 같은 Scene 내 사진 이동 (capturedAt 순)
- 사진이 카메라롤에서 사라진 경우(`isMissing=true`): placeholder + "원본 사진 없음" + 메모/라벨/♥는 표시·편집 가능

### 6.3 상태 / 피드백 비주얼 규칙

- favorite 지정: ♥ 채움 + 상위 슬롯/Spine 마디 크로스페이드 전환 (즉각 피드백)
- 미정리 슬롯: 옅은 회색 + "고르기" 힌트
- 정리 완료 Day: 헤더에 체크 + 진행도 바 반영
- 중복 묶음: 슬롯 모서리에 "겹친 카드" 스택 아이콘

## 7. 기술 스택

### 7.1 iOS Native (Swift)

- **언어**: Swift 5.9+
- **UI**: SwiftUI (iOS 17+ 기준)
- **사진 접근**: PhotoKit (`PHAsset`, `PHAssetCollection`)
- **이미지 분석**: Vision framework (Live Photo 대표컷, 블러 감지, 얼굴 감지 등 기본)
- **온디바이스 ML**: Core ML (v1.5부터 커스텀 모델용)
- **로컬 저장**: SwiftData (라벨/메모/favorite 메타데이터)
- **PDF Export**: PDFKit
- **위치/날짜 기반 그루핑**: CoreLocation reverse geocoding

### 7.2 메타데이터 클라우드 동기화 (v1부터)

- **백엔드**: Apple **CloudKit** (Private Database)
  - 이유: iOS 네이티브 통합, 사용자 iCloud 계정 자동 사용, 추가 서버 비용 0, 사용자 동의 이미 확보
- **동기화 대상**: 라벨, 메모, favorite, trip 정의(기간/이름/표지)
- **동기화 제외**: **사진 원본 (PHAsset의 localIdentifier만 저장)**
- **장점**: 멀티 기기 큐레이션 일관성 + 사진은 기기에 머무름
- **제약**: 사용자가 iCloud Photos를 끈 경우 다른 기기에서 사진 원본 없음 (메타만 있고 사진 없는 상태) — 안내 처리 필요

### 7.3 기능별 기술 난이도

| 기능 | 수단 | 난이도 |
| --- | --- | --- |
| 일자/장소별 묶기 | PhotoKit EXIF + GPS | 쉬움 ✅ |
| 잘 나온 사진 추천 | iOS 18 `VNCalculateImageAestheticsScoresRequest` | 중간 |
| 라이브포토 베스트샷 | Apple 기본 제공 | 직접 개발 X |
| 흔들림 / 블러 감지 | 전용 API 없음 → 커스텀 Core ML / Metal Performance Shaders | 난이도 있음 (v1.5) |
| CloudKit 메타 동기화 | CloudKit + SwiftData (Apple 1st-party 통합) | 중간 |

### 7.4 외부 의존성 & 비용

- 외부 SDK는 분석/크래시 리포트(Sentry 또는 Firebase Crashlytics) 정도만
- 백엔드 서버 없음 → **CloudKit Private Database 사용 (Apple 호스팅, 무료 할당량 내 운영)**
- Apple Developer Program **$99/년**만 필요

## 12. 검증해야 할 핵심 가설 (Phase 0)

1. **타이밍**: 인스타 메이커 사용자는 여행 후 며칠~몇주 안에 사진을 정리·업로드하는가?
2. **자동 vs 수동의 균형**: "자동 1차 완성 → 직접 다듬기" 메시지가 매력적인가, 아니면 한쪽으로 더 기울여야 하는가?
3. **수동 라벨링 의지**: 사용자가 라벨링을 직접 할 의지가 있는가? *(없다면 v1.5의 AI 라벨링을 v1로 당겨야 함)*
4. **Export 우선순위**: PDF 사진첩 vs 카메라롤(인스타용) 중 어느 것을 더 많이 쓸까?
5. **구독 의향**: 무료 trip 개수 제한이 가장 효과적인 conversion 트리거인가?
6. **CloudKit 동기화의 효용**: 사용자가 다기기 사용을 실제로 하는가? (안 한다면 v1에서 CloudKit 비활성화 옵션 고려)

## 13. 디자인 원칙

- **여행 = 1개의 단위**: 모든 흐름이 trip에서 시작
- **삭제 강요 X**: 못생긴 사진도 풍경 좋으면 남길 수 있음. 그룹핑만 함
- **자동 먼저, 사용자는 다듬기만**: 초기 화면은 이미 정리되어 있어야 함
- **재미있게**: 스와이프, 카드 뷰 등 인스타/Tinder 류 UX
- **메모가 1급 시민**: "캘리포니아 카페에서 무엇을 마셨는지"가 사진보다 중요할 수도 있음
- **즉각 피드백**: favorite 지정 시 Spine 마디 + 슬롯 대표컷 즉시 동기화
- **사진은 기기에, 메모는 클라우드에**: 사진 원본은 디바이스 밖으로 나가지 않는다
